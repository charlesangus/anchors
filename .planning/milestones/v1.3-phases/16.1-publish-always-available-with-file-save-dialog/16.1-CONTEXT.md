# Phase 16.1: Publish Always Available with File Save Dialog - Context

**Gathered:** 2026-03-17
**Status:** Ready for planning

<domain>
## Phase Boundary

Fix the Publish button in PrefsDialog so it is always enabled and always shows a file save dialog. Previously the button was greyed out unless `ANCHORS_SITE_CONFIG` was set, blocking site admins from using it to initially create the config file. This phase removes that restriction: the dialog is always available, pre-fills with the best-known path, and persists the chosen path to prefs for future use.

</domain>

<decisions>
## Implementation Decisions

### Button behavior

- Button is always enabled — no more `setEnabled(bool(self._publish_path))` guard
- Button label is always `"Publish..."` (ellipsis indicates a dialog follows — standard Qt convention)
- Every click opens a `QFileDialog.getSaveFileName` dialog — there is no direct-write path

### Dialog initial path (priority order)

1. `ANCHORS_SITE_CONFIG` env var, if set and non-empty
2. `prefs.last_publish_path`, if set (persisted from a previous successful publish)
3. Filesystem root (`"/"`) as the cold-start fallback when neither of the above is available

### Path persistence after publish

- After a successful publish (user confirmed the dialog and write succeeded), store the chosen path to a new prefs field: `last_publish_path` (default `""`)
- Persisted via `prefs.save()` — follows the same save-on-OK pattern as other prefs fields
- This means `_on_publish_naming` calls `prefs_module.last_publish_path = chosen_path` then `prefs_module.save()` after a successful write

### Publish scope

- Unchanged from Phase 16: `prefs.publish(path)` writes only the three naming fields (`naming_regex`, `naming_template`, `naming_demo_filename`)
- `plugin_enabled`, `link_classes_paste_mode`, `custom_colors` remain user-only and are never written to the site config file

### Claude's Discretion

- Exact `QFileDialog.getSaveFileName` filter (e.g., `"JSON files (*.json)"`)
- Whether to show a confirmation message after a successful publish
- Whether to strip the `last_publish_path` entry from prefs if it becomes a dead path

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets

- `PrefsDialog.__init__()` (colors.py:561): `self._publish_path = os_module.environ.get("ANCHORS_SITE_CONFIG", "")` — extend to also check `prefs_module.last_publish_path` when env var is absent
- `PrefsDialog._on_publish_naming()` (colors.py:816): already flushes shadow vars and calls `prefs_module.publish(self._publish_path)` — update to use dialog-chosen path instead
- `prefs.publish(path)` (prefs.py): already writes only the three naming fields — no changes needed to this function
- `prefs.save()` (prefs.py): already persists all module-level prefs vars — adding `last_publish_path` follows the same pattern as existing fields

### Established Patterns

- `setEnabled(bool(...))` pattern for button state — remove this guard from the publish button
- Working-copy pattern: prefs module vars are the source of truth; dialog flushes to them on OK
- `os_module.environ.get(...)` pattern for env var reads — used in `__init__`, keep for initial path resolution

### Integration Points

- `prefs.py`: add `last_publish_path = ""` module-level var; load/save it alongside other fields
- `colors.py PrefsDialog._build_ui()`: remove `setEnabled` guard on publish button; change label to `"Publish..."`
- `colors.py PrefsDialog._on_publish_naming()`: replace `self._publish_path` with dialog-chosen path; persist chosen path to `prefs_module.last_publish_path` on success

</code_context>

<specifics>
## Specific Ideas

- Initial dialog path resolution: `self._publish_path or prefs_module.last_publish_path or "/"`
- The `self._publish_path` instance var (set from env var at `__init__`) is still useful as the first-priority source — keep it, just no longer use it to gate the button
- After a successful publish, update `self._publish_path` too so subsequent clicks in the same dialog session default to the same path

</specifics>

<deferred>
## Deferred Ideas

- Dead-path detection: if `last_publish_path` points to a file that no longer exists, warn the user — deferred, out of scope
- Dynamic re-check of `ANCHORS_SITE_CONFIG` during dialog lifetime — already deferred in Phase 15.1
- Auto-setting `ANCHORS_SITE_CONFIG` env var after publish — out of scope; env var is system-level

</deferred>
